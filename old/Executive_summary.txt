Goal
Build a Service-Level Objective (SLO) KPI dashboard in Power BI Premium for a centralized data team inside a large investment bank. Dashboard must scale across many “capabilities” and the underlying “services” they deliver.

1 · Current State (What’s Already in Place)
Area	Detail
Data ingestion	Jira tickets and changelog already land in a SQL database nightly.
Source tables	• jira_snapshot – 44 columns (current ticket state).
• jira_changelog – 16 columns (status-change history).
Full column lists and data types are attached as
jira_snapshot_fields.csv, jira_changelog_schema.csv.
History window	Initial load = last 6 months (parameter should be easily adjustable).
Refresh strategy	Full nightly refresh (already scripted).
Licensing	Power BI Premium capacity available.
KPIs & visuals	• Each service KPI → simple line chart (6-month window).
• Red constant line = target; green line = actuals w/ markers & data labels.
• Above each chart: target value, month-on-month change, six-month average.
• Per capability: line chart of % KPIs met per month.

Config 'source of truth' KPI targets, ticket-type mappings, and status→time-type rules will be maintained via structured Confluence pages, with each capability team having edit access to their own sections. Confluence serves as the business interface where users make changes, which are then automatically synchronized to Power BI configuration tables nightly. Power BI acts as the technical implementation layer that processes these configurations into the dimensional model.
Distribution	• Monthly email of the full report.
• Each capability’s page also published to SharePoint.
(Exact mailing lists and site URLs TBD).
Future-proofing	Code should be easy to re-point to Jira Cloud later.

4 · Outstanding Items for the Agent to Solve
Confluence Integration: Design page templates and permission schemes that allow capability owners to edit only their designated configuration sections while maintaining overall page structure.

Configuration Synchronization Process
Design and implement automated sync between Confluence and Power BI:

Extract configuration from structured Confluence tables using REST API
Transform to Power BI configuration table format
Load via Power Query with full audit trail
Validate configuration integrity and business rules
Handle sync failures with rollback capabilities
Schedule nightly execution with manual trigger option

Dimensional Model
Design a star schema (fact + dimensions) that supports:

Ticket-level status changes → lead, cycle, response calculations.

Service ↔ Capability hierarchy.

Exact KPI Formulas

Define lead, cycle, response time maths using changelog.

Incorporate rules like exclude_weekends, pause_on_waiting, etc. (see mapping examples in the brief).

Publishing & Alerting

Implement monthly PDF/email subscription and optional SharePoint embedding per capability.

Recommend approach for proactive SLO-breach alerts (Power Automate, Data-alerts, etc.).

Rollback / Hot-fix Process

Propose a lightweight, auditable rollback for bad config pushes.

Future Jira-Cloud Switch

Configuration sync abstracted to support multiple business interfaces (Confluence today, potentially ServiceNow/custom apps later)

Abstract ingestion so on-prem vs cloud endpoints can swap with minimal refactor.

Baseline & Success Metrics Capture

Script to calculate current six-month averages so improvement can be shown post-launch.

Non-Functional
